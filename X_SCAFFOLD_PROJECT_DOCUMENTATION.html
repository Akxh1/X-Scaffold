X-SCAFFOLD

Predict–Explain–Act: An AI-Driven Adaptive Scaffolding Framework
for Intelligent Learning Management Systems
Final Year Research Project • BSc (Hons) Software Engineering
Informatics Institute of Technology (IIT) • University of Westminster
Document Version: 1.0 • March 2026
Abstract: X-Scaffold addresses the critical gap in educational AI where prediction, explanation, and intervention exist as disconnected silos. The framework unifies XGBoost-based mastery prediction, SHAP-powered explainability, and LLM-driven adaptive scaffolding into a single closed-loop Predict–Explain–Act pipeline. Achieving 85.25% test accuracy and 97.54% AUC-ROC on mastery classification, the system delivers personalised, Vygotsky-aligned hints at four scaffolding tiers during student examinations. A mixed-methods evaluation with 10 domain and technical experts confirmed the system’s technical soundness, pedagogical value, and usability.

1. The Problem: Three Disconnected Silos

Modern Learning Management Systems in higher education capture vast amounts of student behavioural data but often fail to provide teachers with the actionable insights needed to proactively support at-risk students. Current research in AI-driven educational support is fragmented into three key, yet disconnected, silos:

Silo What It Does The Gap
Silo 1: Predictive Modelling Develops accurate risk prediction models using ML algorithms on student data. Models are “black boxes” — they produce a risk score but cannot explain why. Teachers cannot trust or act on opaque predictions.
Silo 2: Intervention Frameworks Provides teacher-facing dashboards with “actionable analytics” and alerts. Relies entirely on the instructor to manually design and execute the intervention. No automated action is taken.
Silo 3: AI-Driven Tutoring Uses generative AI to produce micro-level interventions like hints for a single problem. Hints are generic and disconnected from the student’s holistic, longitudinal performance profile.
The Central Gap: No existing framework bridges all three silos — where the critical “why” from the predictive model directly informs the “how” of the intervention. X-Scaffold was designed specifically to close this loop.

2. The Solution: Predict–Explain–Act Pipeline

X-Scaffold’s primary contribution is a novel Predict–Explain–Act closed-loop pipeline that synthesises three AI technologies into a single coherent flow. Rather than stopping at a risk score (Silo 1), relying on the teacher to act (Silo 2), or generating generic hints without context (Silo 3), the system connects all three automatically.

How it works: When a student completes a Level Indicator exam, the XGBoost model classifies their mastery level into one of four categories (At-Risk, Developing, Proficient, Advanced). SHAP then explains which specific behavioural features drove that prediction. Finally, these explanations and the student’s mastery profile are fed into Google Gemini, which generates personalised, Vygotsky-aligned scaffolded hints during subsequent mock examinations. The “why” from prediction directly informs the “how” of the intervention.

3. System Architecture

X-Scaffold is implemented as a decoupled, two-tier architecture combining a full-stack web application with an AI microservice:

Web Application Tier (Laravel)

  • Laravel 12 (PHP 8.2) — MVC backend
  • React 19 via Inertia.js — SPA-like frontend
  • MySQL — Relational database
  • Laravel Breeze — Authentication
  • MLPredictionService — Inter-service bridge

AI Microservice Tier (Flask)

  • Flask 3.0 (Python) — REST API
  • XGBoost — Mastery prediction
  • SHAP KernelExplainer — Feature explanations
  • Scikit-learn — Preprocessing & evaluation
  • Endpoints: /predict, /batch_predict, /health

The MLPredictionService class in Laravel manages all communication with the Flask API. It extracts 11 behavioural features from a StudentModulePerformance record, sends them via HTTP POST, receives the prediction + SHAP explanation, stores the results, and passes the XAI output to the hint generation layer. A local fallback calculation (using the LMS formula) activates automatically when the Flask API is unavailable, ensuring the system never fails silently.

4.3. Target Variable: Teacher-Assigned Labels

A critical design decision was the use of teacher-assigned mastery labels as the target variable instead of a formulaic proxy. A domain expert teacher independently reviewed all 81 real student records (exam scores and behavioural metrics) and assigned one of four mastery classifications. This eliminates the circular dependency problem where a formulaic target is derived from the same input features the model is trained on. Agreement between teacher labels and the formulaic LMS was 59.3% (48/81), confirming the teacher target captures pedagogical nuances the formula cannot.

5. Silo 2: Explain — SHAP Explainability

5.1. What is SHAP?

SHAP (SHapley Additive exPlanations) is a game-theoretic approach to explain machine learning predictions. For each student prediction, SHAP calculates the exact contribution of every feature to the final classification, providing:

  • Local Explanations: Why this specific student received this specific prediction
  • Global Explanations: Which features matter most across all students
  • Additive Contributions: Each feature’s precise positive or negative contribution to the prediction

5.2. The SHAP-to-LLM Explanation Bridge

Raw SHAP output (numerical Shapley values and feature importance charts) is not useful to a non-technical teacher or student. X-Scaffold bridges this gap by translating SHAP contributions into natural language summaries that are then fed directly into the LLM prompt. For example:

“Based on SHAP analysis: Positive factors include score percentage (+0.234), avg confidence (+0.089). Areas needing improvement: hint usage percentage (−0.156), tab switches rate (−0.072).”

This means the student receives a hint that is not only personalised but grounded in the specific behavioural factors the model identified as influencing their mastery level. The instructor dashboard also surfaces these explanations visually, showing which features contributed positively and negatively to each student’s classification.

6. Silo 3: Act — Adaptive Scaffolded Hints

6.1. Vygotsky-Aligned Scaffolding

The hint generation system is grounded in Vygotsky’s Zone of Proximal Development (ZPD) theory. Scaffolding intensity is inversely proportional to mastery level: struggling students receive comprehensive step-by-step guidance, while advanced students receive minimal Socratic prompts that challenge deeper thinking.

6.2. Four Scaffolding Tiers

Level 1: Advanced (Socratic)

A single Socratic question pushing the student to think deeper, with brief acknowledgement of their strengths. Maximum 20 words.

Level 2: Proficient (Guiding)

Two guiding bullet points: a nudging question and a key concept to revisit. Does not reveal the answer. Maximum 40 words.

Level 3: Developing (Structured)

Three numbered steps: Recall (key concept), Watch Out (common mistake), Start Here (first action). Maximum 80 words.

Level 4: At-Risk (Step-by-Step)

Full concept explanation with analogies, step-by-step solution guide, the correct answer with justification, and warm encouragement. Maximum 200 words.

6.3. Three Hint Modes

The HintController operates in three distinct modes depending on the data available for the student:

Mode When Used Behaviour
Diagnostic During Level Indicator exam Fixed moderate hints (L3) since data is still being collected. No personalisation yet.
Generic No Level Indicator data exists Brief one-liner hint (2 bullet points). No SHAP context available.
Personalised After Level Indicator is completed Full adaptive scaffolding using SHAP explanations, mastery level, and behavioural profile. This is the core X-Scaffold experience.

6.4. Google Gemini Integration

Hints are generated by the Google Gemini API (model: gemini-2.5-flash-lite). The prompt includes the student’s mastery level, LMS score, SHAP-derived strengths and weaknesses, full behavioural data (11 features), the question text, and the appropriate scaffolding strategy. The API is called with retry logic (3 attempts with exponential backoff) for rate limit resilience. A comprehensive topic-aware fallback hint system activates when the API is unavailable.

7. Data Pipeline & Methodology

7.1. Data Collection

Real-world behavioural data was collected using a purpose-built Mock Exam Application that captures granular student interactions in real time. A custom dataset was used rather than publicly available datasets (e.g., OULAD, EdNet) because the system requires specific real-time behavioural features — such as hint usage patterns, confidence ratings, and per-question time allocation — that are not captured in existing public datasets.

A total of 81 unique student records were collected across two batches (December 2025 – February 2026), each containing 11 behavioural features.

7.2. Synthetic Data Augmentation: Cholesky Decomposition

Since 81 records are insufficient for training a robust multi-class classifier, the dataset was expanded to 2,000 records using Cholesky decomposition — a technique that preserves the exact inter-feature correlation structure observed in the real data.

1
Extract Statistics: Calculate mean, std, and 11×11 Pearson correlation matrix from real data.
↓
2
Cholesky Decomposition: Decompose correlation matrix R into lower-triangular L where LLT = R.
↓
3
Generate & Transform: Draw 2,000 uncorrelated samples from N(0,1), multiply by LT, scale to original distribution.
↓
4
Label Assignment (KNN): Fit K-Nearest Neighbours (K=5, distance-weighted) on real records. Each synthetic sample inherits its nearest real neighbours’ teacher label.

7.3. Reusable Pipeline Design

The data pipeline was intentionally designed as a bootstrapping framework. The system actively persists all student interaction data during use. As the platform is deployed in real classrooms over full semesters, genuinely labelled data (with verified end-of-semester marks as target variable) will naturally accumulate. Future iterations can retrain the model using this real downstream data without re-engineering the pipeline.

10.1. A/B Testing Protocol

All evaluators experienced a within-subjects A/B test:

  • Test A (Control): Generic, non-personalised hints regardless of student performance — representing conventional LMS approaches.
  • Test B (Experimental): Adaptive scaffolded hints powered by the full Predict–Explain–Act pipeline — representing the X-Scaffold contribution.

Expert consensus confirmed that Test B provided noticeably more useful, contextual guidance. Domain experts validated the pedagogical alignment with Vygotsky’s ZPD, and technical experts confirmed the ML pipeline and architecture were correctly implemented.

13. Limitations & Future Work

13.1. Current Limitations

  • Synthetic Data Dependency: The model is trained primarily on synthetically augmented data. While Cholesky preserves correlations, it cannot capture non-linear relationships or rare edge cases.
  • Proxy Target Variable: The LMS formula serves as a fallback proxy. Without full-semester deployment, verified end-of-semester marks are not yet available as ground truth.
  • External LLM Dependency: Reliance on Google Gemini API raises cost, availability, and data privacy concerns for institutional deployment.
  • Evaluation Scope: Expert impressions and A/B testing demonstrations rather than longitudinal classroom deployment with measured learning gains.
  • Single Subject Domain: Tested within a single subject area; generalisation to mathematics, sciences, and humanities requires further investigation.

13.2. Future Enhancements

  • Real Classroom Deployment: Deploy over full semesters to collect genuine data with verified outcomes, progressively replacing synthetic records.
  • On-Premise LLM: Replace Gemini with locally hosted models (Llama 3, Mistral) to eliminate external API dependencies and address data privacy.
  • Dynamic Scaffolding Transitions: Adjust hint difficulty in real time within a single exam session based on ongoing performance.
  • TreeExplainer Migration: Switch from KernelExplainer to TreeExplainer for near-instant SHAP computation during live exam sessions.
  • Longitudinal Impact Study: Conduct a randomised controlled trial measuring pre/post learning gains over a full academic term.
